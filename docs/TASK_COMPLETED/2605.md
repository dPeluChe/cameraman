## Session: Perf & stability audit (2026-05-19)

> Period: 2026-05-19
> Branch: `fix/perf-and-stability-audit` (PR #15, squash `be99d9a`)
> Focus: five perf/stability findings from external code review — audio realtime thread, codec, telemetry memory, PiP throttle, Sendable.

### fix(audio): mic file writes off the real-time tap thread (be99d9a)

- [x] **`MicAudioRecorder.installTap`** — the tap callback runs on the audio engine's real-time thread. `audioFile.write(from:)` there blocked it and starved the engine, surfacing as the recurring `HALC_ProxyIOContext: skipping cycle due to overload` warnings (and audible glitches in long sessions).
- [x] **`copyBuffer(_:)`** — deep-copy of the `AVAudioPCMBuffer` inside the callback. The engine recycles buffer storage as soon as the callback returns; without the copy the async write reads garbage.
- [x] **`writeQueue`** — dedicated serial `DispatchQueue` (`userInitiated` QoS). The tap callback now does `writeQueue.async { audioFile.write(from: processed) }` and returns immediately.
- [x] **`stopRecording`** drains the queue with `writeQueue.sync { }` before returning the URL, so callers never see a truncated mic file.
- [x] Audio processing (noise gate / echo cancellation) still runs synchronously after the copy but before the queue hop — deterministic CPU work, not I/O.

### fix(capture): HEVC when output exceeds 1920x1080 (be99d9a)

- [x] **`CaptureSessionManager.createVideoWriter`** — `useHEVC = width > 1920 || height > 1080`. H.264 High@Auto refused some ultrawide / 4K dimensions, triggering `AVAssetWriter` error `-10877` around frame 70 on a 3440x1440 display.
- [x] **`AVVideoProfileLevelKey`** dropped from the HEVC path (irrelevant); only set on 1080p-or-below.
- [x] Info log emitted at writer creation announcing the effective codec + resolution + framerate.
- [x] **Closes the 🔴 ultrawide blocker** recorded in `TASK_TODO.md` (3440x1440 → writer status `.failed` → 221KB `screen.mov` for a 30s recording).

### perf(telemetry): JSONL streamed line-by-line (be99d9a)

- [x] **`TelemetryParser.loadTelemetryEvents`** — previously `Data(contentsOf:)` + `String(data:encoding:)` + `components(separatedBy: "\n")`. Peak memory ≈ `file_size × 3` (Data + String + Array<String>).
- [x] Replaced with `FileHandle.bytes.lines` async iteration. Decodes one line at a time; peak memory ≈ a single line.
- [x] **`fileSize`** checked via `FileManager.attributesOfItem` before opening — preserves the original `ParserError.emptyFile` behavior without loading anything into memory.
- [x] Malformed lines are skipped (a partially-written tail line is expected when the app is killed mid-recording).
- [x] **28/28 tests** in `TelemetryParserTests` pass after the refactor.

### perf(pip): camera draft throttle 30Hz → 20Hz (be99d9a)

- [x] **`ProjectEditorPiPView.draftThrottle`** lowered to `1.0 / 20.0`. About 33% less compositor work during active drag; visually indistinguishable from 30Hz.
- [x] **Full refactor deferred** — the proper fix is giving `MaskedVideoCompositor` a dynamic camera property so updates skip the `AVMutableVideoComposition` rebuild. Touches every consumer of the custom compositor; an inline comment on the throttle flags this.
- [x] The external reviewer reported "rebuild 60Hz on every move" — incorrect. The existing architecture already includes: `draftCamera` @State (does not republish `editor.project` during drag), throttle in `pushDraftIfNeeded`, 150ms debounce in `PreviewPlayerView`, short-circuit + light path in `PreviewEngine.updateProject`.

### fix(concurrency): RecordingSession `@unchecked Sendable` (be99d9a)

- [x] **`CaptureEngine.RecordingSession`** marked `@unchecked Sendable`. The class has mutable state (writers, timestamps, error flags) but every read/write is funneled through the parent `CaptureEngine` actor.
- [x] Under Swift 6 strict concurrency, passing a non-Sendable class across actor isolation is a hard error. This unblocks it.
- [x] Doc comment documents the invariant: the parent actor serializes all mutations and no public setters exist. If we ever expose mutation from outside the actor, the annotation must be removed and the session re-modeled (e.g. as its own actor).
